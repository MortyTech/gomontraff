# trafficmon — Go + cilium/ebpf traffic billing exporter

This is a CO-RE rewrite of the original Python + BCC traffic-accounting
Prometheus exporter. It preserves the original's functional behavior
end-to-end: same eBPF accounting logic, same TC ingress/egress attachment
model, same map semantics, same metric name/labels/format, same
periodic-drain billing cycle, same graceful cleanup on SIGINT/SIGTERM.

## 1. Project layout

```
ebpf-traffic-monitor/
├── bpf/
│   └── traffic.bpf.c          # eBPF C program (CO-RE, libbpf conventions)
├── internal/
│   ├── ebpf/
│   │   ├── gen.go             # go:generate bpf2go directive
│   │   ├── loader.go          # loads objects, populates LPM trie
│   │   ├── endian.go          # native-byte-order helper (IP <-> u32)
│   │   └── traffic_bpfel.go   # <- generated by bpf2go, not checked in here
│   │   └── traffic_bpfel.o    # <- generated by bpf2go, not checked in here
│   ├── tc/
│   │   ├── attach.go          # clsact qdisc + classic bpf filter attach/detach
│   │   └── protocol.go
│   ├── metrics/
│   │   └── exporter.go        # Prometheus registry + custom text formatter
│   └── collector/
│       └── collector.go       # periodic batch drain -> Prometheus
├── cmd/trafficmon/
│   └── main.go                # wiring, env config, signal handling
├── go.mod
├── Makefile
└── README.md
```

`internal/ebpf/traffic_bpfel.go` / `traffic_bpfeb.go` and their embedded
`.o` payloads are **generated**, not hand-written — see §7. Once generated
and committed (or generated in CI), the final `go build` needs no clang,
no libbpf, and no BCC on either the build or target machine (aside from
clang being needed once, at build time, to compile the `.bpf.c` — the
*target* machine needs nothing).

## 2. The eBPF C program (`bpf/traffic.bpf.c`)

Functionally identical to the original: same two maps (`traffic_map` — a
hash of `u32 IP -> {rx_bytes, tx_bytes}`, and `monitored_subnets` — an LPM
trie of IPv4 prefixes), same Ethernet/IP header bounds checks, same
"look up target IP in the trie, and if present accumulate `skb->len` into
rx or tx" logic.

**Why it changed:** BCC compiles this C at *runtime* on the target host
using an embedded Clang/LLVM, and provides convenience macros
(`BPF_HASH`, `BPF_LPM_TRIE`, `lookup_or_try_init`) that only exist inside
BCC's preprocessor. None of that exists outside BCC, so:

- `BPF_HASH(...)` / `BPF_LPM_TRIE(...)` → replaced with the standard
  libbpf `SEC(".maps")` struct definitions that `bpf2go`/libbpf understand.
- `lookup_or_try_init()` → replaced with the explicit
  lookup → `bpf_map_update_elem(..., BPF_NOEXIST)` → re-lookup pattern,
  which is exactly what BCC's helper does under the hood.
- `#include <uapi/linux/bpf.h>` etc. (BCC's runtime-resolved headers) →
  replaced with `vmlinux.h` (a machine-generated dump of the kernel's own
  BTF-described types) for CO-RE portability, plus `bpf_helpers.h` /
  `bpf_endian.h` from libbpf for `SEC()`, `bpf_htons()`, and friends.

The packet-parsing and accounting logic itself is untouched.

## 3. Go userspace

`cilium/ebpf` replaces BCC's Python bindings. Key pieces:

- **`internal/ebpf`** — loads the compiled BPF object (`bpf2go`-generated
  Go bindings, no runtime compilation), removes the memlock rlimit, and
  populates the LPM trie from `MONITOR_SUBNETS`, mirroring the original
  subnet-loading loop (including the "skip and warn on invalid CIDR"
  behavior).
- **`internal/tc`** — attaches to the interface using the *same* clsact +
  classic BPF-filter model as the original (via `vishvananda/netlink`,
  a pure-Go netlink library), rather than the newer TCX `bpf_link` API.
  This is a deliberate choice: the task requires preserving the *exact*
  existing TC attachment model. If you're starting a new project rather
  than migrating one, `cilium/ebpf/link.AttachTCX` (kernel 6.6+) is the
  more modern equivalent — it uses a proper `bpf_link` so the kernel
  auto-detaches on process exit, removing the need for manual qdisc
  cleanup entirely. That would be a *behavioral* change (no more manual
  `Cleanup()`), which is why this rewrite doesn't adopt it by default;
  it's called out here as the natural next step. See §9.
- **`internal/metrics`** — a dedicated Prometheus registry (like the
  original's `CollectorRegistry()`, not the global default) exposing a
  single `traffic_bytes` gauge with labels `ip`, `direction`, `interface`.
- **`internal/collector`** — the periodic drain loop.

## 4. TC attach/detach (`internal/tc/attach.go`)

Reproduces, 1:1:

| Original (pyroute2)                                              | Go (vishvananda/netlink)              |
|--------------------------------------------------------------------|----------------------------------------|
| `ipr.tc("add", "clsact", idx)`                                     | `Attacher.EnsureClsact()`              |
| `add-filter bpf :1 parent ffff:fff2 direct_action`                 | `Attacher.AttachIngress(fd, name)`     |
| `add-filter bpf :2 parent ffff:fff3 direct_action`                 | `Attacher.AttachEgress(fd, name)`      |
| `ipr.tc("del", "clsact", idx)` (in the signal handler)              | `Attacher.Cleanup()`                   |

EEXIST on qdisc add is tolerated exactly like the original's `if e.code
!= 17: raise`.

## 5. Prometheus exporter (`internal/metrics/exporter.go`)

Same metric name (`traffic_bytes`), same help text, same label set
(`ip`, `direction`, `interface`), same custom registry-per-process model.

The original monkey-patched `floatToGoString` so that byte counts render
as plain integers (`"1048576"`) instead of Go-style floats
(`"1.048576e+06"`). `client_golang`'s `promhttp.Handler` doesn't expose an
equivalent hook, so this rewrite serves `/metrics` with a small hand-written
formatter (`formatFloatLikeGoString`) that reproduces the same four cases
the original handled (`Inf`, `-Inf`, `NaN`, plain integer) instead of
pulling in `promhttp`.

## 6. Periodic collector (`internal/collector/collector.go`)

Every `REFRESH_INTERVAL` seconds:

1. **Batch drain.** Calls `Map.LookupAndDeleteBatch`, the Go equivalent of
   BCC's `items_lookup_and_delete_batch()` — one syscall reads *and*
   removes every current entry, minimizing the window in which a packet
   could land between "read" and "reset" (same intent as the original).
   If the batch API isn't available (older kernel or map type), it falls
   back to a plain iterate+delete loop so the exporter degrades gracefully
   instead of crashing.
2. **Reset + republish.** Calls `GaugeVec.Reset()` (the Go equivalent of
   `TRAFFIC_METRIC.clear()`) so IPs that went idle disappear from
   `/metrics` instead of keeping a stale value, then sets fresh `rx`/`tx`
   gauges for every IP the batch call returned.

## 7. Build instructions

Build-time requirements (build host only):

- Go 1.22+
- clang/LLVM (for compiling `bpf/traffic.bpf.c` to BPF bytecode)
- `bpftool` (to dump `vmlinux.h` from the build host's kernel BTF) —
  only needed once, or whenever you change which kernel struct fields the
  program touches
- libbpf headers (`bpf/bpf_helpers.h`, `bpf/bpf_endian.h`) — e.g.
  `apt install libbpf-dev` or vendor them into `bpf/`

```bash
# 1. One-time: dump this host's kernel BTF into a portable C header.
make vmlinux

# 2. Compile the eBPF program and generate Go bindings (bpf2go).
make generate

# 3. Fetch Go module dependencies (needs network access).
go mod tidy

# 4. Build a single static binary — no clang/libbpf/BCC needed at runtime.
make build
# -> build/trafficmon
```

Run (needs `CAP_NET_ADMIN`/`CAP_BPF`, so typically as root or via
`setcap`):

```bash
sudo MONITOR_INTERFACE=bond0 \
     MONITOR_SUBNETS=188.64.139.0/24,192.168.1.0/24 \
     REFRESH_INTERVAL=30 \
     EXPORTER_BIND_ADDR=0.0.0.0 \
     EXPORTER_PORT=8000 \
     ./build/trafficmon
```

> **Note:** this sandbox has no network access, so `go mod tidy` /
> `go build` could not be executed here to produce a verified binary.
> The code follows the current cilium/ebpf, client_golang, and
> vishvananda/netlink APIs, but please run `go mod tidy && go build ./...`
> in a connected environment before deploying, and pin `go.sum`.

## 9. Summary of every architectural change and why

| Change | Reason |
|---|---|
| BCC → `cilium/ebpf` + `bpf2go` | Eliminates the runtime Clang/LLVM dependency (and BCC package) on target machines; ships one static binary. |
| Runtime-compiled C → CO-RE `.bpf.c` compiled once at build time | Required to drop the LLVM/Clang runtime dependency; CO-RE + `vmlinux.h` keeps one compiled object portable across kernel versions instead of needing per-kernel headers. |
| `BPF_HASH`/`BPF_LPM_TRIE` macros → libbpf `SEC(".maps")` structs | Those macros are BCC-preprocessor-only; libbpf's struct-based map definition is the CO-RE-compatible equivalent bpf2go understands. |
| `lookup_or_try_init()` → explicit lookup/insert/re-lookup | Same effective semantics; the macro doesn't exist outside BCC. |
| pyroute2 netlink calls → `vishvananda/netlink` | Pure-Go netlink library that supports the exact same clsact qdisc + classic `bpf` filter primitives pyroute2 used, so the TC attachment model is unchanged. |
| BCC's `items_lookup_and_delete_batch()` → `cilium/ebpf` `Map.LookupAndDeleteBatch` | Direct equivalent; same one-syscall snapshot+reset semantics, with a manual iterate+delete fallback for older kernels. |
| `prometheus_client`'s monkey-patched float formatting → hand-written `/metrics` formatter | `client_golang` has no public hook for this; a small custom handler reproduces the exact same integer-only output. |
| Python `signal.signal(...)` → `signal.NotifyContext` + explicit `Cleanup()` | Idiomatic Go shutdown via `context.Context`, same net effect: qdisc/filters removed before process exit. |
| Global mutable module state (Python) → `config`/`Loaded`/`Exporter`/`Collector` structs passed explicitly | Avoids package-level globals per the stated Go style requirement; makes the dependency graph explicit and testable. |

### Deliberately *not* changed

- Packet parsing/accounting logic (Ethernet + IPv4 bounds checks, LPM
  lookup against a `/32` key, `__sync_fetch_and_add` on rx/tx).
- Map names, sizes, and types (`traffic_map`: hash, 65535 entries;
  `monitored_subnets`: LPM trie, 256 entries).
- TC attachment model (clsact qdisc, classic `bpf` classifier,
  `direct_action`, ingress on `ffff:fff2`, egress on `ffff:fff3`).
- Metric name/labels/help text and the "clear, then re-set" refresh
  behavior.
- Environment variable names and defaults (`MONITOR_INTERFACE`,
  `MONITOR_SUBNETS`, `REFRESH_INTERVAL`, `EXPORTER_BIND_ADDR`,
  `EXPORTER_PORT`).
